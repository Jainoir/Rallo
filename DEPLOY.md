# Deploying Rallo to the cloud

Target: **Render** free tier (Docker web services, auto-deploy from GitHub) + **Neon** (managed Postgres) + **CloudAMQP** (managed RabbitMQ). Everything is defined in [render.yaml](render.yaml); this guide is the ~15 minutes of account setup only a human can do. No credit card is required by any of the three providers.

**Security note:** Render's free tier gives every service a public URL. Backend services therefore only accept requests stamped with the gateway's `X-Gateway-Secret` header (`GatewaySecretFilter` in each service) — direct calls get a 401. Both secrets are auto-generated by the blueprint; you never handle them. Swagger UI and health endpoints stay publicly readable on purpose.

## 1. Accounts (all free, no card)

- [Render](https://render.com) — sign in with your GitHub account
- [Neon](https://neon.tech) — Postgres
- [CloudAMQP](https://www.cloudamqp.com) — RabbitMQ, "Little Lemur" free plan

## 2. External infrastructure (~5 min)

**Neon:** create one project, then three databases in it: `auth_db`, `checkin_db`, `notification_db`. Keep the connection details page open — you'll need host, user, and password. The JDBC URLs look like:

```
jdbc:postgresql://<neon-host>/auth_db?sslmode=require
jdbc:postgresql://<neon-host>/checkin_db?sslmode=require
jdbc:postgresql://<neon-host>/notification_db?sslmode=require
```

**CloudAMQP:** create a Little Lemur instance and copy its **AMQPS URL** (`amqps://user:pass@host/vhost`).

## 3. Create the stack on Render (~5 min)

1. Render dashboard → **New** → **Blueprint**
2. Connect the `Jainoir/Rallo` repository (the repo must be pushed; render.yaml is at the root)
3. Render shows the five services it will create. If any name is taken, rename it — then update the three `*_SERVICE_URL` values and `API_UPSTREAM` in render.yaml accordingly and push again.
4. It prompts for the `sync: false` values — paste the Neon URL/user/password per service and the CloudAMQP URL for checkin + notify.
5. Apply. First build takes ~10 min (Maven builds inside Docker).

`JWT_SECRET` and `GATEWAY_SHARED_SECRET` are generated by Render and shared across services automatically (`fromService` references) — nothing to do.

## 4. Verify

```bash
curl -s -X POST https://rallo-jainoir-gateway.onrender.com/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"demo","email":"demo@example.com","password":"password123"}'
```

Then open **https://rallo-jainoir-web.onrender.com**, register, create a goal, check in.

Also verify the hardening: `curl https://rallo-jainoir-checkin.onrender.com/api/goals -H "X-User-Id: anyone"` must return **401** — the service refuses traffic that didn't come through the gateway.

## 5. From now on

Every push to `main` auto-deploys all five services (Render watches the repo; the GitHub Actions pipeline independently gates tests and security scans). No tokens or CI secrets needed.

## Notes

- **Cold starts:** free services spin down after ~15 min idle; a sleeping Spring service takes ~30–60 s to wake. Before a demo, warm everything with one request to the frontend and one register/login call. If a first request times out, retry — the service is booting.
- **Why Neon instead of Render's Postgres:** Render's free Postgres is deleted after 90 days; Neon's free tier persists.
- **Notification timing:** Render wakes services on inbound HTTP only, so a sleeping notification-service consumes queued RabbitMQ events when something calls it. Queues are durable — nothing is lost — and the dashboard calls `/api/notifications` on every load, which wakes it. Milestone notifications simply appear on the next page load.
- **Redis is not deployed:** the gateway only needs it for rate limiting, which isn't enabled yet. Add Upstash later alongside the leaderboard feature.
- **Schemas:** Hibernate creates tables on first connect (`ddl-auto: update`); Flyway migrations are on the roadmap.
